# データ基盤とは何か — 登場人物と役割

## まず直感をつかむ

会社のデータを「料理」にたとえてみよう。畑で採れたばかりの野菜（生データ）は、そのままでは食卓に出せない。洗って、切って、味付けして、皿に盛りつけて、はじめて「食べられる料理」になる。**データ基盤とは、この台所と配膳の仕組みをまるごと用意したもの**だ。畑（業務システム）と、料理を待つお客さん（意思決定する人）のあいだに立ち、生データを「使える形」に変えて届け続ける。

台所が散らかっていれば、料理人が変わるたびに味が変わり、誰も注文しなくなる。データ基盤も同じで、仕組みとして整っていないと「作ったのに使われない」「一部の人しか使えない」状態に陥る。これがこのコースで言う "腐る" の入口だ。

## 正確な定義

データ基盤（Data Platform）とは、**組織内に散らばったデータを収集・蓄積・加工・提供し、意思決定や機能に再現性をもって役立てるための、技術とプロセスとオーナーシップの総体**である。単なるデータベースやBIツール一つを指すのではない。次の3つを束ねた「仕組み」を指す。

- **技術**: 保管庫（DWH）、加工パイプライン、可視化ツール
- **プロセス**: どう更新し、どう品質を担保し、どう公開するか
- **人とオーナーシップ**: 誰が責任を持って維持するか

:::insight
データ基盤の本質は「ツール」ではなく「データが価値に変わる流れを、属人性なく回し続ける仕組み」である。ツールはその流れを支える部品にすぎない。
:::

## なぜ必要か

データ基盤がないと、各部署が自分でCSVを抜き、Excelで集計し、独自の計算をする。すると「売上」の数字が会議ごとに食い違う。同じSQLを何十人もが書き直し、誰かが辞めると再現できなくなる。データ基盤は、この**重複・不整合・属人化**をなくし、「一度ちゃんと作れば、皆が同じ正しい数字を使える」状態をつくるために必要だ。

## 価値の流れと登場人物

データは左から右へ、加工されながら価値を増していく。各段階に担い手がいる。

```mermaid
flowchart LR
  S[業務システム / Source] --> I[収集・蓄積]
  I --> T[加工・モデリング]
  T --> M[Marts / 提供層]
  M --> U[BI・分析・機能で利用]
```

| 役割 | 主な責任 | この図での担当 |
|------|----------|----------------|
| データエンジニア | 収集・蓄積・パイプラインの構築運用、信頼性 | S→I |
| アナリティクスエンジニア | 生データを使いやすいモデル（marts）へ整形 | T→M |
| データアナリスト | 提供層を使い、問いに答え示唆を出す | M→U |
| 利用者（ビジネス職など） | 数字をもとに意思決定・施策実行 | U |
| データオーナー | あるデータ領域の定義・品質・公開可否に責任を持つ | 全段にまたがる |

:::tip
**アナリティクスエンジニア**は比較的新しい役割で、「エンジニアほど低レイヤーではないが、アナリストより再利用可能な資産（dim/fctテーブルなど）を作る」中間の専門職。データエンジニアとアナリストの分断を埋める接着剤だと考えるとよい。
:::

具体例として、利用者が「国別の完了注文額を知りたい」と言ったとする。アナリティクスエンジニアが整えた提供層があれば、アナリストはこう書くだけで済む。

```sql
-- 提供層(marts)があると、生のorder_itemsを毎回JOINせずに済む
SELECT
  c.country,
  SUM(f.amount) AS total_amount
FROM fct_orders AS f
JOIN dim_customer AS c
  ON f.customer_key = c.customer_key
WHERE f.status = 'completed'
GROUP BY c.country
ORDER BY total_amount DESC;
```

提供層がなければ、利用者ごとに `orders` と `order_items` を直接JOINし、`status` の解釈も人によってバラつく。提供層は「皆が同じ材料を同じ味付けで使える」状態をつくる。

## よくあるアンチパターン

:::antipattern
**「ツールを入れれば基盤ができる」と思い込む**。高価なDWHやBIを導入しても、誰がデータの定義に責任を持つか（オーナーシップ）と、どう更新・公開するか（プロセス）が無いと、結局みんなが我流でデータを抜き、数字が割れる。技術だけでは基盤は立たない。
:::

:::warning
役割を一人に押し付けるのも危険だ。収集も加工も分析も定義もすべて一人が抱えると、その人がボトルネックになり、辞めた瞬間に基盤が止まる。役割は「分業して見える化」してこそ続く。
:::

## 腐らせないポイント

このレッスンは失敗モード **1（作ったけど使われない）** と **2（価値がチームに閉じる）** に直結する。

- **使われない（unused）を防ぐ**: 価値の流れの終点は必ず「利用者の意思決定」。技術側の都合ではなく、利用者が何に使うかを起点に設計する。終点の利用が描けない基盤は最初から腐っている。
- **閉じる（siloed）を防ぐ**: 役割とオーナーシップを明示し、提供層（marts）という「皆が使える共通の材料」を用意する。特定の人の頭の中にしか作り方がない状態を避け、誰でも同じ数字にたどり着ける民主化された状態をめざす。

## 演習

問1: 共通スキーマを使い、カテゴリ別の「完了注文の明細金額（quantity × unit_price）」合計を求めるSQLを書け。提供層がまだ無い前提で、生データ（orders, order_items, products）から書くこと。

解答例:

```sql
SELECT
  p.category,
  SUM(oi.quantity * oi.unit_price) AS total_amount
FROM orders AS o
JOIN order_items AS oi
  ON o.order_id = oi.order_id
JOIN products AS p
  ON oi.product_id = p.product_id
WHERE o.status = 'completed'
GROUP BY p.category
ORDER BY total_amount DESC;
```

問2: 上のクエリを「複数の利用者が毎週書く」状況を想像し、どの登場人物が、何を用意すれば重複と数字のブレを防げるか、一文で答えよ。

解答例: アナリティクスエンジニアが、注文明細粒度の提供層テーブル（例: 明細ごとの amount と status を持つ fct テーブル）を用意すれば、各利用者は JOIN と status 解釈を繰り返さずに済み、数字のブレも防げる。

## まとめ

- データ基盤とは、生データを価値に変える流れを属人性なく回し続ける「技術・プロセス・オーナーシップの総体」であり、単なるツールではない。
- 必要な理由は、重複・不整合・属人化をなくし、皆が同じ正しい数字を使えるようにするため。
- 価値はSource→収集→加工→提供→利用と流れ、各段にデータエンジニア／アナリティクスエンジニア／アナリスト／利用者／データオーナーが立つ。
- 設計は技術都合でなく「利用者の意思決定」を起点にすると、使われない失敗（unused）を防げる。
- 役割とオーナーシップを明示し共通の提供層を整えることで、価値がチームに閉じる失敗（siloed）を防げる。
